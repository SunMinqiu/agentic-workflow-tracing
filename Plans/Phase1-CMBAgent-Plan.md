Companion to `Overview.md` §4 (target systems / orchestration-fixedness spectrum) and
`Plans/Phase1-2-Metrics-Attribution-Plan.md` (workflows table). This document designs the
integration of **CMBAgent** (github.com/CMBAgents/cmbagent) as a fifth target system. Nothing

`Plans/Phase1-Plan.md`.

## 1. What CMBAgent is (confirmed by reading the source, not just the README)

CMBAgent is a general-purpose multi-agent scientific research system built on **AG2**
(formerly AutoGen), pip-installable (`cmbagent[all]`, Python 3.12+). Despite the cosmology
name it is domain-agnostic; relevant to us, it also ships a `[materials]`/`[biochem]`/`[data]`
extras split, similar in spirit to how GenoMAS/ChemGraph target one science domain each.

Two entry points matter for tracing:

- **`cmbagent.one_shot(task, agent='engineer', engineer_model=..., work_dir=...)`**
  (`cmbagent/workflows/one_shot.py`) — a single task handed to one agent (default: `engineer`),
  no planning phase, `max_rounds`/`max_n_attempts` bound retries. This is the cheapest,
  simplest entry point — closest in shape to a single ReAct turn (comparable to a one-tool-call
  SRAgent/ChemGraph invocation) and should be the **first** thing traced.
- **`cmbagent.planning_and_control(...)`** (`cmbagent/workflows/planning_and_control.py`) — the
  full pipeline: a `planner` agent and `plan_reviewer` agent iterate to produce an ordered step
  list, then a `control`/`controller` agent executes the plan step-by-step, delegating each step
  to `engineer` / `researcher` / domain-specialized agents (`cmbagent/agents/{planning,control,
  coding,research,specialized}/`). This is the mode that matters most for the paper's
  orchestration-fixedness axis (see Overview §4): the plan itself is dynamically generated per
  run, but execution of a *given* plan is comparatively fixed — a third point on the spectrum,
  distinct from both GenoMAS's static action-unit list and SRAgent/ChemGraph's per-turn
  supervisor loop with no separate planning artifact.

### 1.1 Code execution and file layout (confirmed from `cmbagent/cmbagent.py`, `base_agent.py`)

- The `engineer` agent's code executor is AG2's **`LocalCommandLineCodeExecutor`**
  (`cmbagent/base_agent.py:4,240`) unless a `custom_executor` is supplied (e.g. the
  `RemoteWebSocketCodeExecutor` used by the web UI — irrelevant to us; we always use the local
  executor). This means: LLM writes a code block -> AG2 writes it to a file under
  `work_dir/codebase/` -> subprocess executes it with the local Python interpreter. Structurally
  identical to how GenoMAS/ChemGraph agents' generated code gets executed, so the existing
  eBPF+lineage pipeline applies with no new tracer work.
- `work_dir` (default `work_dir_default`, overridable) is populated with fixed subdirectories on
  every `solve()`/`one_shot()` call (`cmbagent/cmbagent.py:680-693`):
  - `work_dir/data/` — `database_path`, default `"data"`; input datasets.
  - `work_dir/codebase/` — `codebase_path`, default `"codebase"`; **this is where the
    engineer's generated code is written and executed** (also appended to `sys.path`).
  - `work_dir/chats/` — chat/plan transcripts.
  - `work_dir/time/` — per-step timing records.
  - `work_dir/cost/` — `cost_report_<timestamp>.json` (`cmbagent.py:527-531`), a
    **cost/token report CMBAgent already emits itself**.
  - `clear_work_dir` defaults to `False` in `one_shot()` — repeated runs accumulate in the same
    `work_dir` unless the caller clears it, which matters for the reread/reuse metrics (§6.2).

This `data / codebase / chats / time / cost` layout is a cleaner starting point than any other
target system's ad hoc directories: it already separates input, generated-code execution,
provenance, and cost by construction, and maps almost directly onto the path classes
`Plans/Phase1-Plan.md` §3.5 wants (`dataset / tool_outputs / logs / …`).

### 1.2 A built-in provenance signal the other four targets don't have

CMBAgent already writes its **own** per-run `time/` and `cost/*.json` records — real, structured
timing/cost ground truth generated by the framework itself, not something this project's tracer
has to reconstruct from tool-call logs. This should be treated as a second, independent
provenance source for Phase 2 attribution (§3.2 below), alongside the AG2-side hook we still
need to build. If its step boundaries line up with eBPF timestamps, it's a stronger
time-window-matching anchor than the logger-based approach used for SRAgent/ChemGraph, and worth
cross-validating against.

## 2. Orchestration-fixedness classification (for Overview §4)

Already written into `Overview.md` §4 as part of this update: CMBAgent's `planning_and_control`
mode sits **between** GenoMAS (fixed action-unit DAG) and SRAgent/ChemGraph/SciLink-autonomous
(fully dynamic per-turn routing) — plan *generation* is dynamic, plan *execution* is
comparatively fixed once generated. `one_shot` mode is closer to the dynamic end (single
LLM-driven turn, no separate plan artifact). Both modes are worth tracing because they occupy
different points on the axis within the same codebase — a controlled comparison
`Plans/Phase1-Plan.md`-style axes (GEO fanout, cohort scaling) cannot offer.

## 3. What needs to be built

### 3.1 Adapter (`src/agent_io_tracing/adapters/cmbagent/`) **[⬜ TODO]**

Mirror the existing adapter shape (`adapters/chemgraph/launcher.py`,
`adapters/genomas/launcher.py`): a `launcher.py` module invoked as
`python -m agent_io_tracing.adapters.cmbagent.launcher <work_dir> <log_dir> --mode {one_shot,planning_and_control} --task "..." --model ...`
that:

1. `os.chdir`s into a run-scoped `work_dir` (same STOP/CONT handshake pattern as
   `trace_script_bcc_chemgraph.sh` — launch the agent process stopped, start the BCC tracer,
   `CONT` once the tracer signals ready via the FIFO).
2. Calls `cmbagent.one_shot(...)` or `cmbagent.planning_and_control(...)` with `work_dir` pointed
   at the run's output directory so `data/codebase/chats/time/cost` land under the traced dir,
   not some unrelated default.
3. **No LangChain callback handler exists for AG2** — unlike ChemGraph/SRAgent
   (`adapters/sragent/logger.py`'s `LangChainToolLogger`), CMBAgent's agent framework is AG2, so
   the existing `langchain_tool_logger.py`-style hook does **not** apply as-is. Two options, not
   mutually exclusive:
   - (a) **Minimal path**: don't build an AG2-side logger at all for the first pass; rely purely
     on eBPF + CMBAgent's own `time/`/`cost/*.json` files (§1.2) for phase/tool-call windows.
     Cheapest to ship, but coarser join granularity than the per-tool-call logger the other four
     targets have.
   - (b) **Full parity path**: write a small AG2 hook (AG2 agents expose message/reply hooks —
     check `autogen.ConversableAgent.register_hook` at build time against the installed AG2
     version) that emits the same schema as `genomas_tool_logger.py`/`langchain_tool_logger.py`
     (`run_id, agent_id, step_id, parent_step_id, phase, tool name/cwd/pid/exit`) so CMBAgent
     slots into the existing `phase1_metrics.py`/`attribution.py` pipeline with zero changes to
     shared code. **Recommended target**, but start with (a) to get a cheap end-to-end smoke run
     working first (same "validate cheaply before scaling" instruction as `Phase1-Plan.md` §7
     applied to GenoMAS).
4. Flush any handler and exit with the subprocess's exit code, same contract as the other
   adapters.

### 3.2 Provenance / phase tagging **[⬜ TODO]**

CMBAgent-specific phase tags to add once 3.1(b) exists (or reuse `time/`/`cost/` as a cheaper
substitute per 3.1(a)):
- `plan_review_iteration` (planner ↔ plan_reviewer back-and-forth before control starts) —
  this is CMBAgent's structural analogue to GenoMAS's `action_unit_backtrack`: it is where a
  differently-behaved agent would iterate a different number of times for the identical task,
  making it a natural agent-induced (5.1) candidate.
- `control_step:<step_number>` (one per plan step, delegated to whichever agent the plan named)
  — analogous to GenoMAS's per-action-unit tagging; this is the join key for retry-induced I/O
  and reread attribution once built (mirrors `compute_reread_attribution`'s `tool_call_id` join
  in `phase1_metrics.py`, substituting `step_number` for `tool_call_id`).
- `code_exec:<step_number>` for the engineer's `LocalCommandLineCodeExecutor` invocations,
  matching the existing `code_exec:<AgentName>` convention used for GenoMAS.

### 3.3 Config + trace script + deploy script **[⬜ TODO]**

Follow the ChemGraph pattern exactly (`config/config_chemgraph.env`,
`scripts/trace_script_bcc_chemgraph.sh`, `scripts/deploy_chemgraph_to_client.sh`):

- `config/config_cmbagent.env` — `CMBAGENT_REPO` (pip install, not a git checkout, unless
  pinning a commit for reproducibility — recommend pinning via `pip install
  cmbagent==<version>` in the deploy script for the same reproducibility reason
  `deploy_chemgraph_to_client.sh` pins `CHEMGRAPH_GIT_REF`), `CMBAGENT_MODEL`
  (start with a cheap OpenAI model, same reasoning as `Plans/Phase1-GEO-Fanout-Plan.md` §0 —
  e.g. `gpt-4o-mini` or a `gpt-5-nano` tier model), `WORKLOADS` array of `name|mode|model|prompt`.
- `scripts/trace_script_bcc_cmbagent.sh` — same STOP/CONT/FIFO handshake, same
  parse→summarize→parallelism→viz post-processing pipeline (`agent_io_tracing.parsing.ebpf`,
  `.analysis.summary`, `.analysis.parallelism`, `.viz.trace` — these are workload-agnostic and
  need zero changes).
- `scripts/deploy_cmbagent_to_client.sh` — `pip install "cmbagent[all]"` (or a narrower extra
  matching the demo task's domain, e.g. `[astro]` for a CAMB-based task) into a venv on
  Lustre/scratch, smoke-check `import cmbagent`, write `.env.cmbagent` with
  `AGENT_PYTHON`/`TRACER_PYTHON`/`CMBAGENT_MODEL`, same shape as
  `deploy_chemgraph_to_client.sh` §[4/5].

### 3.4 First workload prompts (cheap, for pipeline validation before scaling) **[⬜ TODO]**

Per `Plans/Phase1-Plan.md` §7's validation-before-scaling instruction, start with the cheapest
possible smoke task in `one_shot` mode before attempting `planning_and_control`:
- `one_shot` smoke: a trivial engineer task with no external data dependency (e.g. "draw two
  random numbers and print their sum, saving the result to a file in the work directory") —
  mirrors the README's own one-liner example, validates the adapter/tracer plumbing at near-zero
  API cost.
- `planning_and_control` smoke: a small multi-step task that forces at least 2-3 plan steps
  across different agents (e.g. a small data-analysis task using the `[data]` extra) to confirm
  the plan/control phase split is actually observable in the eBPF+phase trace before committing
  to a `[astro]`/CAMB-based task that would pull in heavier dependencies and runtime.

## 5. Open action items (in build order)

1. Write `src/agent_io_tracing/adapters/cmbagent/launcher.py` for **`one_shot` mode only** first
   — smallest surface area, no planning phase to get wrong on the first attempt.
2. Write `config/config_cmbagent.env`, `scripts/trace_script_bcc_cmbagent.sh`,
   `scripts/deploy_cmbagent_to_client.sh`, reusing the ChemGraph scripts as templates almost
   verbatim (same STOP/CONT/FIFO/post-processing shape).
3. Run the `one_shot` smoke workload (§3.4) once, locally or on the existing CloudLab node,
   confirm `phase1_metrics.json` populates with non-degenerate values the way the GenoMAS/
   ChemGraph cells did, **before** writing any CMBAgent-specific phase-tagging code.
4. Only then extend the adapter to `planning_and_control` mode and decide between the 3.1(a)
   minimal-provenance path and the 3.1(b) full AG2-hook path — informed by whether CMBAgent's own
   `time/`/`cost/*.json` files turn out to be sufficient join keys on the smoke run's real data,
   rather than deciding this up front.
5. Update `Plans/Phase1-2-Metrics-Attribution-Plan.md`'s workflows table with the actual cell
   count once real runs exist (currently: none — do not mark CMBAgent as counted until a run
   produces a real `phase1_metrics.json`, per that document's "what counts as collected" rule).
